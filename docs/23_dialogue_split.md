# 長いDialogueEvent分割 方針整理

## 1. 目的

長いセリフがYouTubeショート画面から切れる問題について、どの段階で分割するべきかを整理する。

第46回では、長い字幕が画面から切れる問題への最小対応として、ASS字幕本文に `\N` を入れる簡易折り返しを実装した。`--ass-wrap-chars` / `--ass-max-lines` は短めの字幕には有効だが、実際の長いセリフでは2行を超える部分が最後の行にまとまり、結局画面外に切れることが分かった。

この問題は、単なる表示調整ではなく、イベント設計の問題でもある。表示の問題だと思ったら、実際には「1つのDialogueEventが長すぎる」ことが根本原因だった、という見方が必要になる。

本仕様書では、ASS側だけで字幕を分割するのではなく、VOICEVOXに投げる前に長い `DialogueEvent` を短い `DialogueEvent` へ分割する方針を整理する。

現時点では、単体の `DialogueEvent` を分割する `split_long_dialogue_event` に加えて、イベント列全体へ分割を適用する `split_long_dialogue_events` を追加している。ただし、CLI引数や既存の音声生成パイプラインへの組み込みはまだ行っていない。

## 2. なぜASS側の折り返しだけでは不十分か

`--ass-wrap-chars` / `--ass-max-lines` は、1つの字幕を2行程度に収めたい場合には有効である。短めの字幕であれば、ASS本文に `\N` を入れるだけで画面内に収まりやすくなる。

しかし、2行を超える長文では限界がある。現在の簡易折り返しでは、`max_lines` を超えた分を最後の行にまとめるため、長いセリフでは2行目が非常に長くなり、画面外に切れる可能性が残る。

ASS側で1つの長い字幕を複数の `Dialogue:` 行に分割すること自体はできる。ただし、1つの音声の中で、どの文字が何秒時点で読まれているかは分からない。たとえば、20秒の音声を3つの字幕に分ける場合、単純に文字数比で時間を割ることはできるが、実際の発話速度、間、抑揚、VOICEVOXの読み上げ結果とはずれる可能性がある。

そのため、ASS側だけで複数Dialogueへ分割すると、字幕と音声の同期は近似になる。短い確認用途なら許容できる場合もあるが、投稿用動画の品質を上げるには不安が残る。

## 3. 本命はVOICEVOXに投げる前の分割

本命の方針は、長い `DialogueEvent` を音声生成前に短い `DialogueEvent` へ分割することである。

音声生成前に分割すれば、分割後のセリフごとにVOICEVOXへ投げられる。すると、それぞれのWAVファイルの長さが分かり、そのWAV長から自然にSRT時刻を作れる。

この方式では、字幕の単位と音声の単位が一致する。したがって、ASS側で無理に字幕だけを分割するより、音声と字幕の同期が自然になる。

イメージ:

```txt
長いDialogueEvent
  ↓
短いDialogueEventに分割
  ↓
各DialogueEventをVOICEVOXで音声生成
  ↓
各WAVの長さからSRT時刻を生成
  ↓
all.wav と output.srt を生成
```

この方針では、既存のSRT生成やWAV連結の仕組みを、分割後のイベント列に対して使える。字幕焼き込み側は、従来どおりSRTからASSを生成すればよい。

## 4. どこで区切るかが難しい

分割の難しさは、どこで区切るかにある。

単純に文字数だけで切ると、不自然な字幕になりやすい。変なところで切ると字幕が読みにくくなるだけでなく、VOICEVOXに渡す文章も不自然になり、発声や間に影響する可能性がある。

悪い例:

```txt
設計書を作って、実
装して、テストして
```

良い例:

```txt
設計書を作って、
実装して、テストして、
```

つまり、文字数は目安として使いつつ、意味の切れ目を優先する必要がある。最初から高度な自然言語処理を入れるのではなく、句読点などの分かりやすい記号を使ったルールベースの分割から始めるのが現実的である。

## 5. 分割ルール案

まずは高度な自然言語処理ではなく、ルールベースで考える。

候補位置の優先順位:

1. 句点 `。` で切る。
2. 読点 `、` で切る。
3. 感嘆符 `！`、疑問符 `？` で切る。
4. スペースや明示的な区切りがあれば使う。
5. どうしても無理なら文字数で切る。

基本方針:

- `max_chars` を超える `DialogueEvent` だけを分割対象にする。
- できるだけ `max_chars` 以内で、後ろに近い区切り候補を選ぶ。
- 区切り記号は前側の文に含める。
- 細切れになりすぎないように、分割後の最小文字数 `min_chars` を考慮する。
- `min_chars` 未満の短すぎる断片ができる場合は、前後の断片に寄せる。
- 区切り候補がない場合だけ、文字数ベースで切る。

この段階では、自然な文節分割、文字幅ベースの調整、音素タイミング解析までは行わない。

## 6. 分割条件案

将来のCLI候補として、以下を検討する。

```txt
--split-long-dialogue
--dialogue-split-chars
--dialogue-split-min-chars
```

### --split-long-dialogue

長い `DialogueEvent` を音声生成前に分割するかどうかを指定する。

短期的にはデフォルト無効にして、手動確認しながら有効化するのが安全である。既存の台本から生成される音声や字幕の単位が変わるため、いきなり常時有効にすると既存出力との差分が大きくなる。

### --dialogue-split-chars

1つの `DialogueEvent` の目安文字数を指定する。

たとえば、ショート動画では1字幕あたり14〜20文字程度を目安にする可能性がある。ただし、実際にはフォントサイズ、画面幅、字幕位置、2行表示の見た目によって調整が必要になる。

### --dialogue-split-min-chars

分割後の短すぎる断片を避けるための最小文字数を指定する。

たとえば、`max_chars=18` で分割するときに、末尾に2〜3文字だけ残るような分割は避けたい。短すぎる断片は読みづらく、VOICEVOXの発声単位としても不自然になりやすい。

今回は、これらのCLIは実装しない。仕様上の候補として整理するだけにする。

## 7. 既存機能との関係

既存の台本フォーマットは変更しない。

分割対象は `DialogueEvent` のみとする。SEや間イベントは分割対象外であり、イベント列の中でそのまま保持する。

分割時に引き継ぐ情報:

- 話者
- 字幕テキスト
- 読み上げテキスト
- スタイル指定
- 将来的な `speed` / `pause` などの発話パラメータ

SRT生成とWAV連結は、分割後の `DialogueEvent` 列に対して行う。つまり、既存パイプラインの前段でイベント列を整えるイメージである。

```txt
台本ファイル
  ↓
DialogueEvent列
  ↓
長いDialogueEventを分割
  ↓
音声生成
  ↓
WAV連結 / SRT生成
```

この方式なら、分割後の各WAV長から字幕時間を作れるため、音声と字幕の同期を保ちやすい。

## 8. 読み上げ || 字幕 形式への注意

現在の台本フォーマットでは、読み上げと字幕を分けられる可能性がある。

例:

```txt
めたん：こんにちはなのです || こんにちは。
```

この場合、単純に読み上げテキストだけを分割すると、字幕との対応が崩れる可能性がある。逆に字幕だけを分割しても、VOICEVOXへ渡す読み上げ単位とは一致しない。

短期的な案:

### 案A: 読み上げと字幕が同一の場合だけ自動分割する

読み上げテキストと字幕テキストが同じ場合だけ、自動分割する。分割後も読み上げと字幕の対応が崩れにくい。

### 案B: 読み上げと字幕が分離されている場合は自動分割対象外にする

読み上げと字幕が分離されている `DialogueEvent` は、自動分割しない。手動で台本側を分けるか、将来のより高度な対応に回す。

推奨は案Bである。まずは、読み上げと字幕が分離されている `DialogueEvent` は自動分割対象外にする。対応関係が崩れるリスクを避け、実装の責務を小さく保てる。

短期的な推奨方針としては、読み上げと字幕が同一の `DialogueEvent` だけを、VOICEVOX投入前にルールベースで分割する。

## 10. 次に実装するなら

次に実装するなら、実装済みのイベント列分割をCLIまたは音声生成前処理へ接続することを検討する。

```python
split_long_dialogue_events(events, max_chars, min_chars) -> list[ScriptEvent]
```

方針:

- `DialogueEvent` のみ対象にする。
- 長さが `max_chars` 以下ならそのまま返す。
- `max_chars` を超える場合、句点・読点などの候補位置を探す。
- `min_chars` 未満の短すぎる断片を作らない。
- 話者、スタイル、`speed` / `pause` などの将来パラメータは引き継ぐ。
- SEや間イベントはそのまま保持する。
- イベント順序を維持する。
- 分割後のイベント列を、既存の音声生成・SRT生成パイプラインに渡す。

単体の `DialogueEvent` 分割とイベント列全体への適用は実装済みである。次の段階では、まだ未実装のCLI引数や既存パイプラインへの接続を検討する。

## 11. テスト方針

実装済み、または将来実装する場合のテスト観点を整理する。

- 短い `DialogueEvent` は分割されないこと。
- 長い `DialogueEvent` が句点で分割されること。
- 句点がない場合、読点で分割されること。
- 句点・読点がない場合、感嘆符や疑問符で分割されること。
- 区切り候補がない場合、文字数で分割されること。
- 短すぎる断片を作らないこと。
- 話者情報が維持されること。
- スタイル指定が維持されること。
- 将来的な `speed` / `pause` などの発話パラメータが維持されること。
- `DialogueEvent` 以外は分割されないこと。
- SEや間イベントの位置が維持されること。
- イベント順序が維持されること。
- 読み上げ `||` 字幕の分離があるイベントは、自動分割対象外にすること。
- `split_long_dialogue_events` では、イベント列全体に適用しても順序が維持されること。

## 12. 短期的な推奨方針

短期的には、まず読み上げと字幕が同一の `DialogueEvent` だけを、VOICEVOX投入前にルールベースで分割する。

これにより、分割後のセリフごとにWAVが生成され、それぞれのWAV長からSRT時刻を自然に作れる。ASS側で字幕だけを無理に分割するより、音声と字幕の同期を保ちやすい。

ASSの `--ass-wrap-chars` / `--ass-max-lines` は、分割後の字幕をさらに画面内へ収めるための補助として残す。根本的には、長すぎる発話単位を前段で短くする方針を優先する。
