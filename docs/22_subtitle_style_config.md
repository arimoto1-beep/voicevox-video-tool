# 字幕スタイル設定 方針整理

## 1. 目的

字幕スタイル設定を、今後どのように調整・管理していくかを整理する。

現在は、ASS字幕スタイルを `make_video.py` 内の固定値として持っている。この方式は初期実装としては単純で扱いやすいが、手動確認を重ねると、字幕サイズや位置の試行錯誤のたびにコードを書き換える必要がある。

第42回の手動確認では、スタイル変更が実際の動画に反映されることは確認できた。一方で、`short` / `normal` ともにまだ見た目の調整余地があり、固定値を直接変更するだけでは調整サイクルが重くなってきている。

そのため、短期的な調整のしやすさ、中期的な設定管理、将来的な話者ごとの字幕スタイルや立ち絵表示への拡張を分けて考える。

## 2. 現在の状況

現在の実装では、`make_video.py` 側でSRTを読み込み、ASSを生成している。

現在の流れ:

```txt
background.png
all.wav
output.srt
  ↓
make_video.py
  ↓
output.ass
  ↓
output_subtitled.mp4
```

ASS字幕スタイルは、`make_video.py` 内の `get_ass_subtitle_style(layout)` がlayoutごとの固定値として返している。

現在の値:

### short

```txt
font_name: Yu Gothic UI
font_size: 96
margin_v: 260
outline: 5
shadow: 1
alignment: 2
```

### normal

```txt
font_name: Yu Gothic UI
font_size: 72
margin_v: 150
outline: 4
shadow: 1
alignment: 2
```

## 3. 第42回の手動確認で見えた課題

### short

- 前よりは改善した。
- ただし、ショートとしてはもっと中央寄りでもよさそう。
- 文字サイズもまだ少し小さく感じる。
- 長い字幕では改行位置が問題になりそう。

### normal

- 位置はこのくらいでもよさそう。
- 文字はやや小さめ。

## 4. 選択肢

字幕スタイルを調整しやすくする方法として、以下の選択肢がある。

### 案A: 固定値を再調整する

概要:

`make_video.py` 内の初期値をもう一度変更する。

メリット:

- 実装が簡単。
- 既存構造をほぼ変えない。
- テスト更新も少ない。

デメリット:

- 試行錯誤のたびにコード修正が必要。
- 複数パターンを比較しづらい。
- 話者ごとのスタイルには向かない。

### 案B: CLI引数で一時指定する

例:

```powershell
python make_video.py `
  --audio tmp/cli_pipeline/all.wav `
  --background assets/background_short.png `
  --srt tmp/cli_pipeline/output.srt `
  --output tmp/video/output_short_subtitled.mp4 `
  --layout short `
  --ass-font-size 112 `
  --ass-margin-v 420 `
  --ffmpeg .\ffmpeg\bin\ffmpeg.exe
```

メリット:

- 試行錯誤しやすい。
- コードを変更せずに見た目を確認できる。
- 短期的なチューニングに向いている。

デメリット:

- 引数が増える。
- layoutごとの管理がやや煩雑。
- 話者ごとのスタイルには向かない。
- 設定がコマンドに散らばる。

### 案C: 設定ファイルで管理する

例:

```yaml
subtitle:
  layouts:
    short:
      font_name: "Yu Gothic UI"
      font_size: 112
      margin_v: 420
      outline: 5
      shadow: 1
      alignment: 2

    normal:
      font_name: "Yu Gothic UI"
      font_size: 84
      margin_v: 180
      outline: 4
      shadow: 1
      alignment: 2
```

メリット:

- layoutごとの設定を管理しやすい。
- コードを変えずに設定を変更できる。
- 将来的に話者ごとの設定へ拡張しやすい。
- 立ち絵や背景など、他の動画レイアウト設定とも統合しやすい。

デメリット:

- 設定ファイル読み込み処理が必要。
- バリデーションが必要。
- 依存ライブラリを増やすかどうか検討が必要。
- 初期実装としては少し重い。

### 案D: プリセットを用意する

例:

```txt
--subtitle-preset short_large
--subtitle-preset short_safe
--subtitle-preset normal_large
```

メリット:

- CLIが比較的シンプル。
- よく使う設定を名前で切り替えられる。
- 試行錯誤用にも使いやすい。

デメリット:

- プリセット定義をどこに持つか決める必要がある。
- 自由な調整には向かない。
- 結局、設定ファイル化と近い話になる可能性がある。

## 5. 話者ごとの字幕スタイル問題

将来的には、ずんだもん・めたんなど、発言者ごとに字幕色やスタイルを変えたい可能性がある。

ただし、現時点のSRTには話者情報が残っていない。SRTから得られる情報は以下だけである。

- 開始時刻
- 終了時刻
- 字幕本文

話者ごとのスタイルには、少なくとも以下のような情報が必要になる。

```txt
start_sec
end_sec
speaker
text
```

現在の `parse_srt_file` からASSを作る方式では、`speaker` がないため、話者ごとのスタイル分けは難しい。本文に話者名を埋め込んで解析する方法も考えられるが、字幕本文とメタ情報が混ざるため、長期的な構造としては扱いにくい。

将来的には、台本イベントまたは字幕キューに `speaker` を持たせる必要がある。理想は、話者情報つき字幕キューからSRTとASSの両方を生成する構造である。

イメージ:

```txt
台本イベント
  ↓
話者情報つき字幕キュー
  ├─ output.srt
  └─ output.ass
```

この構造にすると、SRTは互換用・確認用として出力しつつ、ASSでは話者ごとの色、縁取り、位置、立ち絵表示などへ拡張しやすくなる。

## 6. 設定ファイル案

設定ファイル化する場合、まずはlayoutごとの字幕スタイルを扱う想定にする。

例:

```yaml
subtitle:
  default:
    font_name: "Yu Gothic UI"
    font_size: 96
    margin_v: 260
    outline: 5
    shadow: 1
    alignment: 2

  layouts:
    short:
      font_size: 112
      margin_v: 420

    normal:
      font_size: 84
      margin_v: 180
```

この形では、`default` に共通値を置き、`layouts.short` や `layouts.normal` で必要な値だけを上書きする。重複を減らしつつ、layoutごとの調整を明確にできる。

将来的には、話者ごとの設定も持てる可能性がある。

例:

```yaml
subtitle:
  speakers:
    ずんだもん:
      primary_color: "&H00FFFFFF"
      outline_color: "&H00000000"

    めたん:
      primary_color: "&H00FFE0FF"
      outline_color: "&H00000000"
```

ただし、話者ごとの設定を有効にするには、字幕キュー側に `speaker` が必要である。設定ファイルだけを先に用意しても、現在のSRT入力からは話者を判別できない。

## 7. 短期方針

短期的には、まず設定ファイル化を急がない。

短期対応として、CLI引数で `font_size` と `margin_v` を一時指定できるようにする。目的は、コードを書き換えずに字幕サイズと位置を確認できるようにすることである。

長い字幕が画面外に切れる問題への最小対応として、ASS本文だけを文字数ベースで簡易折り返しできるようにする。SRTは確認用・互換用として変更せず、ASS焼き込み時のDialogue本文にだけASS改行 `\N` を入れる。

ただし、CLI引数が増えすぎるのは避ける。必要なら、まずは以下の2つ程度に絞る。

```txt
--ass-font-size
--ass-margin-v
--ass-wrap-chars
--ass-max-lines
```

layout別のデフォルト値は今のまま持つ。CLI引数が指定された場合だけ、選択されたlayoutのデフォルトスタイルに対して一時的に上書きする。

`--ass-font-size` / `--ass-margin-v` を省略した場合は、従来どおりlayout別デフォルト値を使う。`--srt` が指定されない字幕なし動画生成では、これらの値は実質的に使われない。
`--ass-wrap-chars` を省略した場合は、従来どおり折り返しを行わない。`--ass-max-lines` の標準値は2行とし、最大行数を超えた分は最後の行にまとめる。

短期対応の範囲では、自然な文節分割、句読点優先、文字幅考慮、行間調整、話者ごとの字幕スタイル、設定ファイル管理、プリセットまでは扱わない。

## 8. 中期方針

中期的には、字幕スタイルの設定ファイル化を検討する。

扱う内容:

- layoutごとのsubtitle styleを設定ファイルで管理する。
- `default` 値とlayout別上書きを分ける。
- 設定値のバリデーションを行う。
- 未指定の値は既存のデフォルトで補完する。

設定ファイル形式の候補は、JSONまたはYAMLである。

JSON:

- Python標準ライブラリだけで読み込める。
- 依存ライブラリを増やさずに済む。
- コメントを書けないため、人間が調整する設定としては少し読みづらい。

YAML:

- 人間が読み書きしやすい。
- コメントを書きやすい。
- 使う場合は `PyYAML` などの依存ライブラリが増える。

まずはJSONから始める案もある。字幕スタイル設定の構造が固まってから、必要に応じてYAMLへ移行するか検討する。

## 9. 将来方針

将来的には、話者情報つき字幕キューを導入する方針が望ましい。

SRTだけでは、話者情報、立ち絵、表示位置、話者ごとの演出などを扱うには限界がある。SRTは確認用・互換用として残しつつ、内部データとしては台本イベントに近い情報を保持した字幕キューを使う方がよい。

将来的に扱いたいこと:

- 話者ごとの字幕スタイル
- 話者ごとの字幕色
- 話者ごとの立ち絵表示
- 話者ごとの立ち絵位置
- 簡易口パク
- 台本イベントとの連携
- overlay画像との連携

この段階では、ASS生成だけでなく、動画全体の構成情報として、誰が、いつ、何を話しているかを扱う必要がある。

## 10. 今回まだ実装しないこと

短期対応後も、以下はまだ実装しない。

- 設定ファイル読み込み
- 話者ごとの字幕スタイル
- 話者情報つき字幕キュー
- プリセット
- 自然な文節分割
- 行間調整
- 立ち絵
- 口パク
- overlay画像
- 高度な複数行字幕制御

## 11. 次に実装するなら

次に拡張するとしたら、以下の案がある。

### 案1: CLI引数で試作用パラメータを追加する

候補:

```txt
--ass-font-size
--ass-margin-v
--ass-wrap-chars
--ass-max-lines
```

目的:

コードを変更せずに字幕サイズ、位置、簡易折り返しを試せるようにする。

この案は短期対応として実装する。第42回の手動確認で見えた「shortをもっと中央寄りにしたい」「文字サイズをもう少し大きくしたい」という調整を、コード変更なしで試せるようになる。

### 案2: JSON設定ファイルを追加する

候補:

```txt
--style-config config/subtitle_style.json
```

目的:

layoutごとの字幕スタイルを外部ファイルで管理する。

短期対応としては案1の方が小さく、現実的である。設定ファイル化は、調整したい項目が増えた段階、またはlayout以外の動画設定もまとめて管理したくなった段階で検討する。

## 12. テスト方針

将来実装する場合のテスト方針を整理する。

### CLI引数案の場合

- `parse_args` で `ass_font_size` / `ass_margin_v` / `ass_wrap_chars` / `ass_max_lines` が受け取れること。
- 指定時にデフォルトスタイルを上書きできること。
- 未指定時は従来のlayout別デフォルト値になること。
- `build_ass_content` のStyle行に上書き後の値が反映されること。
- `build_ass_content` のDialogue行に簡易折り返し後の `\N` が反映されること。
- `--srt` 未指定時の字幕なし動画生成は従来どおりであること。

### 設定ファイル案の場合

- 設定ファイルを読み込めること。
- layoutごとの設定を解決できること。
- `default` 値とlayout別上書きをマージできること。
- 未指定値は既定値で補完されること。
- 不正な設定でエラーになること。
- 設定ファイル未指定時は従来のlayout別デフォルト値になること。
- `build_ass_content` のStyle行に解決後の値が反映されること。
